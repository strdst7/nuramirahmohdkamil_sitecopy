**Step 3 — Create src/lib/contentful.ts:**
Create a server-side Contentful client following the existing singleton pattern from `src/lib/supabase.ts`. Implement:

1. **Client singleton** — lazy-init `contentful.createClient()` using `process.env.CONTENTFUL_SPACE_ID`, `process.env.CONTENTFUL_DELIVERY_API_KEY`, and `process.env.CONTENTFUL_ENVIRONMENT || "master"`.

2. **`fetchArticles(): Promise<Article[]>`** — fetches all entries of content type `article` (Contentful content type ID). Maps each entry's fields to the Article interface:
   - `id` ← `entry.sys.id`
   - `title`, `date`, `excerpt` ← string fields
   - `body` ← rich text → extract plain text using a `richTextToPlainText()` helper (concatenate all text node values; for the listing page, body is shown as plain text, not rendered rich)
   - `tags` ← array field (short-text list)
   - `iconName`, `diagramAlt` ← string fields
   - `diagramSrc` ← media field → `entry.fields.diagramSrc.fields.file.url` (with `https:` prefix if protocol-relative; use `fields.file.url` not `fields.file.details.image`)

3. **`fetchProjects(): Promise<Project[]>`** — fetches all entries of content type `project`. Maps fields similarly:
   - `id` ← `entry.sys.id`
   - `number`, `title`, `description` ← string fields
   - `tags` ← array field
   - `iconName`, `imageAlt` ← string fields
   - `imageSrc` ← media field → `entry.fields.imageSrc.fields.file.url` (with `https:` prefix)
   - `cta` ← optional object field with `label` and `href` strings
   - `layout` ← string field (`"right"` or `"left"`)

4. **`getArticleSlugs(): Promise<string[]>`** — fetches all article entries and returns an array of slugs. Generate slug from the article `id` field (lowercase, replace spaces/special chars with hyphens). Also add the raw `id` as a slug variant so existing ids like `"obs-04-12"` work.

5. **`fetchArticleBySlug(slug: string): Promise<Article | null>`** — fetches a single article by matching its `id` field to the slug (after normalizing). Returns the full Article entity with `body` preserved as the raw Contentful rich text `Document` object (not plain-text extracted — the detail page will render it). Returns `null` if no match.

6. **Fallback wrapper** — wrap each fetch function in a try/catch. On any error (network, auth, 5xx from Contentful), fall back to the hardcoded `src/data/` arrays:
   - `fetchArticles()` fallback: `import { articles } from "@/data/insights"` (dynamic import inside catch, or import at top with a flag)
   - `fetchProjects()` fallback: `import { projects } from "@/data/projects"`
   - `getArticleSlugs()` fallback: map over fallback articles' `id` fields
   - `fetchArticleBySlug()` fallback: find in fallback articles array by matching `id` to slug
   
   Log the error via `console.error("[contentful] fetchArticles failed, using fallback:", error)` before returning fallback data. This gives observability in Vercel logs.

7. **Type for rich text body** — export a `RichTextDocument` type alias for the Contentful rich text JSON structure (use `import type { Document } from "@contentful/rich-text-types"`).

**Per D-05:** ISR configuration happens in the page components (`export const revalidate = 3600`), not in the client utility. The client utility is a pure data-fetching layer — it does not set ISR config.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Contentful CDN → Page render | Contentful API responses enter the application at fetch time (server-side only) |
| env vars → Contentful client | `CONTENTFUL_DELIVERY_API_KEY` crosses from deployment environment into the client initialization |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-01 | Information Disclosure | Contentful client initialization | mitigate | `CONTENTFUL_DELIVERY_API_KEY` accessed exclusively via `process.env.*` in `src/lib/contentful.ts` (server-only). No `NEXT_PUBLIC_` prefix. Key never reaches client bundle — enforced by Next.js server component boundary. |
| T-05-02 | Denial of Service | Contentful API fetch | mitigate | All fetch functions wrapped in try/catch with fallback to `src/data/` hardcoded files. If Contentful returns 5xx, the page still renders usable content. `console.error` logs the failure for observability. |
| T-05-03 | Spoofing | Contentful API responses | accept | Contentful Delivery API returns published content only — no user-generated content flows through this path. Content authors are trusted internal users. Content tampering would require Contentful account compromise (out of scope for this application layer). |
| T-05-04 | Tampering | Rich text content rendering | mitigate | Rich text is rendered server-side via `@contentful/rich-text-react-renderer` with a closed set of allowed node types (per D-03: headings, bold, italic, lists, dividers, blockquotes only). Embedded entries and inline embeds are not rendered, reducing XSS surface. |
| T-05-05 | Elevation of Privilege | Contentful API key scope | mitigate | Use Contentful Delivery API key (read-only) — never the Preview API key or Management API key. Delivery key only grants read access to published content. Documented in `.env.example` with the key name `CONTENTFUL_DELIVERY_API_KEY` to distinguish from management keys. |
</threat_model>
