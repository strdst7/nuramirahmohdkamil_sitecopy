Create `src/app/api/leads/route.ts` with a POST handler following the Phase 3 route pattern (NextRequest import, try/catch with Response.json error shape):

```typescript
import { NextRequest } from "next/server";
import { z } from "zod";
import { supabase } from "@/lib/supabase";

// Zod schema per D-05, D-02 field spec, D-11 column types
const leadSchema = z.object({
  project_name: z
    .string()
    .min(1, "Project name is required")
    .max(200, "Project name must be under 200 characters"),
  email: z
    .string()
    .email("Invalid email address")
    .optional()
    .or(z.literal("")),
  whatsapp: z
    .string()
    .regex(/^\+?[\d\s\-()]{7,20}$/, "Invalid WhatsApp number")
    .optional()
    .or(z.literal("")),
  intent: z.enum(
    ["white-label", "update-notification", "beta-access"],
    { errorMap: () => ({ message: "Intent must be one of: white-label, update-notification, beta-access" }) }
  ),
});

export async function POST(request: NextRequest) {
  try {
    // 1. Parse and validate input (per D-06: validate on submit, not on blur)
    const body = await request.json();
    const parsed = leadSchema.safeParse(body);

    if (!parsed.success) {
      const fieldErrors = parsed.error.flatten().fieldErrors;
      return Response.json(
        { error: "Validation failed", fieldErrors },
        { status: 400 }
      );
    }

    const { project_name, email, whatsapp, intent } = parsed.data;

    // 2. Verify env vars exist (Phase 3 pattern: check before using)
    if (!process.env.SUPABASE_URL || !process.env.SUPABASE_SERVICE_ROLE_KEY) {
      return Response.json(
        { error: "Supabase configuration missing — check environment variables" },
        { status: 500 }
      );
    }

    // 3. Insert into playground_leads (per D-09, D-11)
    const { data, error } = await supabase
      .from("playground_leads")
      .insert({
        project_name,
        email: email || null,       // D-11: nullable
        whatsapp: whatsapp || null, // D-11: nullable
        intent,
      })
      .select()
      .single();

    if (error) {
      console.error("Supabase insert error:", error);
      return Response.json(
        { error: "Failed to save lead — database error" },
        { status: 500 }
      );
    }

    // 4. Return success with created record (200, not 201 — Phase 3 convention)
    return Response.json({ success: true, lead: data }, { status: 200 });
  } catch (error) {
    console.error("POST /api/leads error:", error);
    const message =
      error instanceof Error ? error.message : "Unknown server error";
    return Response.json({ error: message }, { status: 500 });
  }
}
```

**Key implementation details:**
- `safeParse` (not `parse`) to avoid throwing — all errors return structured JSON
- `.or(z.literal(""))` on optional fields so empty strings from HTML forms pass as absent values
- WhatsApp regex: accepts international format (+60...), spaces, dashes, parens; rejects obviously invalid
- Import style mirrors Phase 3: `import { NextRequest } from "next/server"` (not `NextApiRequest`)
- Error shape mirrors Phase 3: `{ error: string, fieldErrors?: Record<string, string[]> }` or `{ error: string }`
- Response uses `Response.json()` (Next.js 16 convention) not `NextResponse.json()`
- `console.error` for server-side logging (no structured logger — simple is sufficient per Phase 3)

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → POST /api/leads | Untrusted form data crosses from browser to server |
| POST /api/leads → Supabase | Server-side insert into database with service role key |
| process.env → src/lib/supabase.ts | API keys must never reach client bundle |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01 | Spoofing | POST /api/leads | mitigate | Zod `safeParse` validates all 4 fields server-side — rejects invalid types, empty required fields, bad intent enum values. Client validation is cosmetic only. |
| T-04-02 | Tampering | POST /api/leads | mitigate | Data parsed via `request.json()` + Zod schema on every field — no raw body insertion. Supabase JS SDK uses parameterized queries — no SQL injection risk. |
| T-04-03 | Information Disclosure | src/lib/supabase.ts | mitigate | `SUPABASE_URL` and `SUPABASE_SERVICE_ROLE_KEY` accessed exclusively via `process.env` in server-only module. Module is never imported in any `"use client"` file. Bundle audit during build verifies zero key leakage. |
| T-04-04 | Denial of Service | POST /api/leads | accept | No rate limiting in v1 (deferred per CONTEXT.md). Vercel Edge provides basic DDoS protection. Acceptable for portfolio site with low traffic expectations. |
| T-04-05 | Elevation of Privilege | POST /api/leads | mitigate | No RLS or auth on playground_leads table — service role key used server-side only. Client never obtains any Supabase key. Intent field constrained by Zod enum — cannot inject arbitrary values. |
</threat_model>
