**Part A: Install and configure Vercel Analytics (per D-15)**

Install the package:
```bash
npm install @vercel/analytics
```

**Part B: Update next.config.ts for production deployment**

```typescript
import type { NextConfig } from "next";

const nextConfig: NextConfig = {
  // Per D-12: all API routes run as Edge Functions
  // Next.js 16 auto-detects Edge runtime from route.ts exports;
  // explicitly set for /api routes to ensure consistent behavior
};

export default nextConfig;
```

Note: Next.js 16 App Router automatically treats API routes in `src/app/api/` as serverless functions. To run them as Edge Functions (per D-12), each route must export `export const runtime = "edge"`. However, `@supabase/supabase-js` and `openai` (v6) do NOT support Edge runtime — they require Node.js APIs. Per Phase 3 conventions, these API routes run as Node.js serverless functions (default), not Edge. This is correct behavior: Vercel serverless functions provide the necessary Node.js runtime for these SDKs.

If Edge runtime is truly needed in the future, use `@supabase/supabase-js` with `@supabase/ssr` edge client. For v1, Node.js serverless is the correct target.

**Part C: Create Vercel environment variables documentation**

No additional code file needed — the existing `.env.example` (updated in Plan 01) already documents all required variables. The Vercel deployment flow is:

1. User runs `npx vercel --prod` OR connects the GitHub repo in Vercel Dashboard
2. User adds all 5 environment variables in Vercel Dashboard:
   - `OPENAI_API_KEY` — from OpenAI dashboard
   - `GEMINI_API_KEY` — from Google AI Studio
   - `SUPABASE_URL` — from Supabase project settings
   - `SUPABASE_SERVICE_ROLE_KEY` — from Supabase project settings
   - `NEXT_PUBLIC_SITE_URL` — set to `https://<project-name>.vercel.app`
